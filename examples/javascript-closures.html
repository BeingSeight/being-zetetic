─── PHASE 1 ─── ❓ What is a function, in the most basic sense? → A function is a named sequence of instructions that can be invoked (called) any number of times, with different inputs, and may return a value. → The word function comes from Latin functio — "a performing, an execution". A function is something that performs when called. The inputs are the arguments. The output is the return value. The list of instructions inside is the body. Think of a function as a recipe. The arguments are the ingredients. The body is the steps. The return value is the finished dish. You can run the same recipe with different ingredients and get different dishes. ➜ NEXT QUESTION: "When a function runs, where do its arguments and local variables live?" ─── PHASE 2 ─── ❓ Where do a function's variables live while it runs? → In a region of memory called a stack frame. → A stack frame is created the moment a function is called and destroyed the moment it returns. The frame holds all the function's local variables, plus the arguments it was called with. When the function returns, the frame is gone — and so are all the variables it contained. The name "stack" comes from the data structure: each new function call pushes a new frame on top, and when a function returns, its frame is popped off. Last in, first out. Think of a stack of plates. Each plate is a function call. When the function finishes, you take its plate off the stack and throw it away. ➜ NEXT QUESTION: "If the frame is destroyed when the function returns, what happens to a variable that an inner function tries to use after the outer function has already returned?" ─── PHASE 3 ─── ❓ What happens to a variable that an inner function uses after the outer function has returned? → Normally, it would be gone. But if the inner function still holds a reference to that variable, the variable is not destroyed. It survives. That surviving variable-and-function pair is a closure. → JavaScript does this because, unlike C or Java, it does not destroy local variables the moment their function returns. It only destroys them when no function can reach them anymore. As long as one function still references a variable, that variable lives. The word closure means "closed over" — the inner function "closes over" the outer variables it references, carrying them with it like a backpack. ➜ NEXT QUESTION: "If the inner function is the only thing still holding a reference to the outer variable, how does the program know to keep the variable alive?" ─── PHASE 4 ─── ❓ How does JavaScript know to keep a variable alive? → By reachability. The JavaScript engine constantly asks: "Starting from the root (the global object, the current call stack), which values can I reach?" Any value that can be reached is kept alive. Any value that cannot be reached is collected as garbage. → The inner function is reachable (someone has a reference to it). The variable is reachable through the inner function (the inner function's code references it). So the variable is reachable. So it stays in memory. Think of the engine as a janitor with a flashlight. Every few seconds, the janitor walks the whole program starting from the roots. Anything the flashlight touches stays. The rest gets thrown out. ➜ NEXT QUESTION: "If the inner function and the variable are both still alive, where do they live? On the stack or somewhere else?" ─── PHASE 5 ─── ❓ Where do closures live in memory? → On the heap, not the stack. → The stack is for short-lived things tied to a function call. Closures can outlive the function that created them, so they cannot live on the stack — the stack frame is gone. Instead, the engine allocates the closure's "backpack" (the captured variables) on the heap, where it can persist as long as something still references it. The heap is the long-term parking lot of memory. The stack is the workbench. Closures, because they outlive their creator, belong on the parking lot. ➜ NEXT QUESTION: "If a closure can keep a variable alive forever, can that cause problems — like a program that keeps using more and more memory?" ─── PHASE 6 ─── ❓ Can closures cause memory leaks? → Yes, if you accidentally keep a reference to a large object that should have been collected. → A classic example: putting event listeners on a button inside a function that creates a closure capturing a large array. The button still has the listener. The listener still references the closure. The closure still references the array. The array never gets collected, even after the user navigates away. The fix is always the same: break the chain. Remove the listener. Set the reference to null. Anything that lets the janitor's flashlight lose sight of the value. ➜ NEXT QUESTION: "Beyond the memory question, why are closures useful — what can you do with them that you couldn't do before?" ─── PHASE 7 ─── ❓ What are closures actually good for? → Three things, in order of how often you'll use them: 1. Data privacy. A function can return another function that has access to variables nobody else can see. This is how you build "private" variables in JavaScript. 2. Partial application. A function can return a new function with some arguments already filled in. This is how libraries like Lodash let you write _.partial(fn, a). 3. Callbacks that remember. A function passed to setTimeout or addEventListener can remember variables from where it was created, not just where it's called. The deep answer: closures let you carry state with behavior. A closure is a function plus the environment it was born in. That combination is what makes functional programming possible in JavaScript. ─── CHECKPOINT ─── 1. In one sentence, what is a closure? 2. Why does the inner function's reference keep the variable alive? 3. What's the difference between the stack and the heap, in the context of closures? 4. Name two practical uses of closures. ────── ANSWERS ────── 1. A closure is a function plus the variables it captured from where it was created — a function with a "backpack". 2. The JavaScript engine keeps any value reachable from the roots alive. If the inner function is reachable, and it references the variable, the variable is reachable too. 3. The stack is for short-lived function-call data. Closures can outlive their creating function, so they live on the heap, which is for long-lived data. 4. Data privacy, partial application, callbacks that remember state, and event handlers that retain context.
