─── PHASE 1 ─── ❓ What is a list? → A list is a sequence of items in a particular order. → The word list comes from Old English liste — "a strip, a border, a row". A list is a row of things. The order is part of what makes it a list; the same items in a different order are a different list. Think of a list as a queue at a coffee shop. The first person in line is served first. The order matters. ➜ NEXT QUESTION: "If a list is just a sequence, what's special about appending an item to it — what does 'appending' actually do?" ─── PHASE 2 ─── ❓ What does "appending" do? → Appending means: take the current list, and produce a new list that is identical except for one extra item at the end. → Crucially, appending does not change the old list. The old list still exists, in its original form, with its original length. The new list is a separate thing that contains everything the old one did, plus the new item. In a notebook, you can't unwrite a page. But you can take a fresh notebook and copy every page from the old one, then add your new page. The new notebook is the appended list. The old one is still around. ➜ NEXT QUESTION: "If we never change old lists and we only append, what does the latest list look like compared to the first one?" ─── PHASE 3 ─── ❓ What does the latest list contain that the first one doesn't? → Every item that was ever appended, in the order it was appended. → This is the key insight: an append-only list is a history. It is the complete record of everything that ever happened, in order. Nothing was edited. Nothing was deleted. Nothing was rewritten. Every fact is preserved exactly as it was added. The word ledger means exactly this: a book of accounts where each entry is added but never altered. A bank's ledger is an append-only list. So is a court record. So is a blockchain. ➜ NEXT QUESTION: "If the list lives in one place — one computer, one notebook — what stops that one place from being dishonest about what it contains?" ─── PHASE 4 ─── ❓ What stops one place from being dishonest about the ledger? → If only one copy exists, nothing. The holder of that copy can change it, deny entries, or fabricate ones that never happened. → The fix is distribution: keep thousands of identical copies of the list on thousands of different computers, run by thousands of different people, in thousands of different jurisdictions. No single one of them can change the list without the other 9,999 disagreeing. Think of a town square announcement board. If only the mayor has a copy, the mayor can rewrite history. If every citizen has a copy and they compare notes weekly, the mayor would have to convince every citizen to rewrite at the same time. ➜ NEXT QUESTION: "If we have 10,000 copies, how do they agree on what's in the list — what's the rule for adding a new entry?" ─── PHASE 5 ─── ❓ How do 10,000 copies agree on a new entry? → Through consensus: a rule, agreed in advance, for what counts as a valid new entry. → The most famous rule is "proof of work": to add a new entry, you must do a small but real amount of computational work that the others can quickly verify. If you cheat, the others reject your entry. If you do the work honestly, the others accept it and add it to their copies. The work is a kind of tax. It costs you time and electricity, but it costs the cheater more than the honest participant. The system is designed so that honesty is cheaper than cheating. ➜ NEXT QUESTION: "When does a new entry become part of the official list — how do we know it stuck?" ─── PHASE 6 ─── ❓ When does a new entry "stick"? → When enough copies have accepted it that reversing the decision would be more expensive than accepting it. → In Bitcoin, the rule is "six confirmations" — once six more entries have been added on top of yours, the cost of rewriting history is so high that the entry is considered final. The metaphor: each new entry is a layer of concrete poured on top of the previous one. After enough layers, the bottom is permanent. ➜ NEXT QUESTION: "Why do we call this whole structure a 'block'chain?" ─── PHASE 7 ─── ❓ Why "block-chain"? → Because the entries are bundled into blocks, and each block contains a reference to the block that came before it. The references form a chain. → Each block holds a few hundred entries. At the end of the block is a pointer — a kind of fingerprint — of the previous block. Change one character in any previous block, and the fingerprint breaks, and every honest copy knows the chain has been tampered with. Think of the chain as a stack of sealed envelopes. Each envelope contains a list of transactions and the seal of the envelope below it. If you break the seal of any envelope, every envelope above it becomes invalid. ➜ NEXT QUESTION: "What does a 'fingerprint' of a block look like, and why is it useful?" ─── PHASE 8 ─── ❓ What is a fingerprint of a block? → A hash — a fixed-length string of characters produced by running the block's contents through a mathematical function. The same input always produces the same hash. Even a tiny change in the input produces a totally different hash. → A good hash function is one-way: you can compute the hash from the input, but you cannot compute the input from the hash. This is what makes a fingerprint useful: you can verify that the input matches the fingerprint, but you cannot forge an input that matches a fingerprint you made up. The word hash comes from the French hacher — "to chop up". A hash chops the input into a fixed-size pulp that cannot be reassembled. ─── CHECKPOINT ─── 1. In one sentence, what is a blockchain? 2. Why is an append-only list more useful than a mutable one? 3. What problem does distribution solve? 4. Why does a hash make a good fingerprint? ────── ANSWERS ────── 1. A blockchain is an append-only ledger, distributed across thousands of computers, where each new bundle of entries ("block") contains a reference to the bundle before it, forming an unbreakable chain. 2. Append-only lists preserve history. Mutable ones let history be rewritten. 3. Distribution removes the need to trust any single holder of the ledger. Trust is replaced by agreement among many. 4. A hash is one-way and tiny changes produce huge changes, so it is a reliable, unforgeable fingerprint of large data.
