─── PHASE 1 ─── ❓ What is the most basic thing a computer does? → It executes instructions, one after another, with absolute precision. → A "computer" in the strict sense is any device that takes a list of operations and performs them in order. The word computer comes from Latin computare — "to sum up, to count". Originally a human job. Think of a computer as a very fast, very literal reader. It does exactly what the next instruction says, and nothing else. If the next instruction is wrong, the computer does the wrong thing faithfully. ➜ NEXT QUESTION: "If the computer just reads instructions, where do those instructions come from — and who decides what to read next?" ─── PHASE 2 ─── ❓ Where do the instructions come from? → From memory. A computer has two kinds: the programs (the instructions themselves) and the data (what the instructions operate on). Both live in memory. → Memory is the computer's working surface. Everything the computer knows how to do, every file you have open, every webpage you visit, every line of code — all of it is, at some moment, sitting in memory as bytes. Think of memory as the desk where the work happens. The CPU is the hand that picks up the next item. Without the desk, the hand has nowhere to put anything. ➜ NEXT QUESTION: "If the CPU just reads the next instruction and does it, and the next instruction is in memory, then who decides which instructions to put in memory, in what order?" ─── PHASE 3 ─── ❓ Who decides what runs, and when? → The operating system. → The OS is a program that runs continuously, and its job is to manage every other program. It decides which program gets the CPU next, which one gets which piece of memory, which one gets to read from the disk, which one gets to talk to the network. The word operating is the key. The OS doesn't just "run" — it operates, the way a manager operates a kitchen: scheduling cooks, distributing ingredients, making sure no two cooks burn down the same stove. ➜ NEXT QUESTION: "If the OS is itself a program, who decides that the OS gets to run, and not some other program?" ─── PHASE 4 ─── ❓ Who loads the OS in the first place? → A small program called the bootloader, which lives in a special, protected region of the disk that the CPU knows to look at the moment it powers on. → The chain is: power on → CPU looks at a fixed address → finds the bootloader → bootloader loads the OS → OS takes over. The bootloader is tiny — often under a megabyte. It only has one job: find the OS on the disk, load it into memory, and hand control to it. After that, the bootloader is gone, and the OS owns the machine. ➜ NEXT QUESTION: "Once the OS is running, programs want to use the hardware — but they can't just talk to the disk or the network directly. Why not?" ─── PHASE 5 ─── ❓ Why can't programs talk to hardware directly? → Three reasons. First, two programs might try to use the same hardware at once. Second, a buggy program could crash the entire machine. Third, you'd have to rewrite every program for every kind of disk, network card, and printer that exists. → The OS solves all three by sitting between programs and hardware. Programs make a system call — a polite, well-defined request — and the OS arbitrates. The system call is the handshake. The program says "I'd like to read this file, please." The OS says "Yes, here's the data" or "No, you don't have permission" or "Wait, another program is reading it first." ➜ NEXT QUESTION: "If the OS is the only thing that talks to hardware, what does it look like inside — how is it organized?" ─── PHASE 6 ─── ❓ How is the OS organized inside? → In two big halves. The kernel runs with full hardware access and is the only part allowed to talk directly to devices. Everything else — the file browser, the settings panel, the command line — is just a regular program that asks the kernel to do things. → The kernel is the only part of the OS that, if it crashes, takes the whole machine down. Everything else is just a program. You can close them. The kernel is always there. Think of the kernel as the manager in the kitchen — always present, always arbitrating, never leaving. The chefs come and go. ➜ NEXT QUESTION: "When multiple programs want the CPU at the same time, how does the kernel decide who runs?" ─── PHASE 7 ─── ❓ How does the kernel share the CPU? → By pretending. The kernel rapidly switches the CPU between programs — thousands of times per second — so that each one appears to have the machine to itself. → This trick is called preemptive multitasking. The kernel keeps a queue of "ready to run" programs. It gives each one a small slice of time (often 10ms), then yanks the CPU away and gives it to the next. The illusion of simultaneity is just fast switching. Like a film projector: 24 still frames per second, and your brain sees motion. 1000 context switches per second, and your programs feel like they're running in parallel. ➜ NEXT QUESTION: "If the kernel is constantly switching between programs, how does each program remember where it was when it gets the CPU back?" ─── CHECKPOINT ─── 1. In one sentence, what is the kernel's job? 2. Why is the bootloader so much smaller than the OS? 3. What is a system call, and why does it exist? 4. In what sense is "multitasking" an illusion? (Answers are below — try to answer before reading.) ────── ANSWERS ────── 1. The kernel is the only part of the OS that talks directly to hardware and arbitrates between programs. 2. The bootloader only has to find and load the OS. Everything else — drivers, services, user interfaces — is part of the OS itself. 3. A system call is a program's polite, well-defined request to the kernel to do something with hardware. It exists so the kernel can arbitrate and protect the machine. 4. Multitasking is the kernel rapidly switching the CPU between programs so fast that each one appears to run continuously.
